## 一、项目概述

### 1.1 项目背景

实现某房地产公司的**房屋销售管理系统**。系统在**命令行终端**中完成楼盘与房屋的信息管理、多条件查询、购买结算（含折扣）等业务；数据通过 **MySQL 数据库** 持久化；采用经典 **四层分层架构**，满足 JAVA 高级编程大作业全部硬性要求，并便于答辩现场演示。

### 1.2 核心目标

| 目标 | 说明 |
|------|------|
| 楼盘管理 | 维护楼盘名称、占地面积、地址、开发商等 |
| 房屋管理 | 维护楼号、房号、面积、单价、总价、销售状态等 |
| 查询功能 | 按楼盘、楼号、面积区间、价格区间、销售状态等查找 |
| 购买功能 | 选房 → 计算总价 → 应用折扣 → 确认成交 → 更新数据库 |
| 数据库 | JDBC 存取，建表脚本见 `sql/` |
| 分层架构 | cli → service → dao → database |
| 终端交互 | 菜单驱动、输入校验、操作反馈清晰 |

### 1.3 非目标（本期不做，违反课程要求）

- Windows 窗体 / Swing / JavaFX
- Java Web / Spring Boot HTTP 接口
- Android、微信小程序等任何非控制台形式
- 仅文件存储、不使用数据库（**不符合要求**）

### 1.4 业务真实性扩展（文档组调研写入报告）

在题目主功能基础上，建议调研并设计：

- **客户信息**：购买时登记客户姓名、电话
- **销售员**：记录经办人（可简化为当前登录名）
- **折扣规则表**：数据库存配置，而非硬编码
- **销售统计**：按楼盘统计在售/已售套数、销售额（答辩加分）

## 二、需求分析

### 2.1 功能需求清单

#### A. 楼盘信息管理

- [ ] 新增楼盘（名称、占地面积、地址、开发商、备注）
- [ ] 修改楼盘信息
- [ ] 删除楼盘（需检查是否有关联房屋）
- [ ] 查看楼盘列表 / 楼盘详情

#### B. 房屋信息管理

- [ ] 新增房屋（关联楼盘、楼号、房号、面积、单价，自动计算总价）
- [ ] 修改房屋信息（未售出状态可改）
- [ ] 删除房屋（仅未售出）
- [ ] 查看房屋列表（可按楼盘筛选）

#### C. 房屋查找

- [ ] 按楼盘名称查找
- [ ] 按楼号查找
- [ ] 按面积区间查找
- [ ] 按价格区间查找
- [ ] 按销售状态查找（在售 / 已售）
- [ ] 组合条件查找（进阶）

#### D. 房屋购买

- [ ] 选择待售房屋
- [ ] 展示房屋详情与原价
- [ ] 应用折扣（会员折扣、满减等，至少 2 种规则）
- [ ] 计算实付金额并确认购买
- [ ] 更新房屋状态为「已售」，写入成交记录表

#### E. 系统辅助

- [ ] 主菜单与二级菜单导航
- [ ] 输入合法性校验
- [ ] 数据库连接与异常提示
- [ ] 演示用初始数据（`sql/init-data.sql`）

### 2.2 数据库表设计

```
building（楼盘）
├── id, name, land_area, address, developer, remark, created_at

house（房屋）
├── id, building_id(FK), building_no, room_no, area, unit_price
├── total_price, status(ON_SALE/SOLD), sold_at

sale_record（成交记录）
├── id, house_id(FK), original_price, discount_type, discount_value
├── final_price, customer_name, sold_at
```

完整 DDL 见 [sql/schema.sql](./sql/schema.sql)。

### 2.3 折扣规则（建议至少 2 种）

| 规则 | 说明 | 示例 |
|------|------|------|
| 固定折扣 | 按折扣率计算 | 9 折 → 0.9 |
| 满额优惠 | 总价超阈值减固定金额 | 满 200 万减 10 万 |
| VIP 等级 | 预设等级对应折扣 | 金卡 95 折 |

## 三、技术方案

### 3.1 技术栈

| 项 | 选型 |
|----|------|
| 语言 | Java 17+ |
| 构建 | Maven |

| 注释 | JavaDoc，规范见 [docs/design/代码注释规范.md](./docs/design/代码注释规范.md) |

### 3.2 分层架构（课程核心要求）

```
┌─────────────────────────────────────┐
│  表示层 (cli)                        │  MenuController, ConsoleUtils
│  职责：菜单、输入输出、调用 Service    │
├─────────────────────────────────────┤
│  业务逻辑层 (service)                │  BuildingService, PurchaseService
│  职责：业务规则、事务逻辑、折扣计算    │
├─────────────────────────────────────┤
│  数据访问层 (dao)                    │  BuildingDao, HouseDao, SaleRecordDao
│  职责：SQL 执行、结果集映射为实体      │
├─────────────────────────────────────┤
│  数据库 (MySQL)                      │  building, house, sale_record
└─────────────────────────────────────┘

横切模块：
  model/    — 实体类，与表字段对应
  config/   — 数据库连接配置 DBConfig
  discount/ — 折扣策略（策略模式）
  util/     — 校验、ID 生成等工具
```

**层间调用规则**：cli 只调 service；service 只调 dao；dao 不处理业务判断。

## 十、原创性要求（避免零分）

两组代码完全一样将 **均为零分**，团队须主动差异化：

| 措施 | 说明 |
|------|------|
| 独立业务设计 | 自研字段（如客户表、折扣规则表）、菜单结构、提示文案 |
| 独立命名 | 包名、类名、方法名、表名团队内统一，勿抄其他组 |
| 独立折扣逻辑 | 至少 2 种自有实现的折扣策略 |
| 独立 SQL 与 dao | 自行编写 JDBC，禁止共享同一份 dao 源码 |
| 分工可追溯 | 每类 @author 不同，与分工表对应，证明各自实现 |
| 代码审查 | 合并前组长检查是否与外组或网络成品高度相似 |
